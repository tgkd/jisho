# Jisho ‚Äî Prioritized Code-Quality, Bug & Database Audit

## Executive Summary

The codebase is broadly healthy and offline-first by design, but the recent Expo SDK 55‚Üí56 / RN 0.85 upgrade (commit `8137c9e`) introduced the most serious regressions and there are **no critical, ship-stopping defects** ‚Äî the ceiling is **six high-severity issues** (see appendix). The dominant themes are: (1) **the entire test suite is non-runnable** ‚Äî `jest-expo` throws on a missing preset and every node DB suite fails to compile because `@types/jest` is out of `ts-jest` scope; (2) **`ColorSchemeName` type unsafety** from RN 0.85 (`'light' | 'dark' | 'unspecified'`) that `?? "light"` no longer narrows, producing ~21 `TS7053` errors and a latent crash across the theming layer; (3) **React render-phase violations** ‚Äî a self-sustaining refetch loop on `kanji/[id]`, render-time ref/closure mutation in practice screens, and a repo-wide `setState`-in-effect lint cluster; (4) **search/DB correctness and query-plan issues** ‚Äî homograph dictionary entries silently dropped by a `PARTITION BY` dedup, and full table scans on the hottest word-detail and kanji-search paths. There is also no `typecheck` script and no CI, so all of the above accumulated unnoticed. Most fixes are small and high-leverage; the test and theming fixes are near one-liners with outsized impact.

## Top fixes (do these first)

1. **Restore the node test suite** ‚Äî add a scoped `tsconfig.test.json` (`types: ["jest","node"]`, `include: __tests__/**`) and point `ts-jest` at it; install `@react-native/jest-preset@0.85.3`. Verified to take `parse.test.ts` from 0 ‚Üí 23/23 passing.
2. **Narrow the color scheme once** ‚Äî replace every `useColorScheme() ?? "light"` with `useColorScheme() === "dark" ? "dark" : "light"`. One pattern fix in `hooks/useThemeColor.ts` (+ `Card.tsx`, `subscription-info.tsx`) clears ~21 `TS7053` errors and the latent crash.
3. **Stop the `kanji/[id]` refetch loop** ‚Äî key the effect on `[params.id]` (not `[params]`) and add an `active` guard; this is a self-sustaining SQLite/CPU/battery drain for the screen's lifetime.
4. **Fix homograph search** ‚Äî remove the `ROW_NUMBER()/row_num=1` `PARTITION BY (word, reading)` dedup in `searchByJapanese`/`searchByEnglish`; it silently hides 567 distinct dictionary entries (e.g. only "mall" for „É¢„Éº„É´).
5. **Disable React Query auto-retry for AI streams** ‚Äî set `defaultOptions.queries.retry = false`; failed remote AI generations currently re-fire up to 3√ó with backoff, burning paid tokens on server-error paths.

## 1) Critical / Bugs

> No critical issues. The entries below are the highest-impact correctness/perf bugs.

- **Search drops distinct homograph entries (`PARTITION BY` dedup).** Severity: **High**. `services/database/search.ts` (searchByJapanese ~139‚Äì160; searchByEnglish ~221‚Äì229). The `ROW_NUMBER() OVER (PARTITION BY w.word, w.reading) ‚Ä¶ WHERE row_num=1` collapses distinct JMdict ids sharing a surface, hiding 567 entries across 498 surfaces (e.g. „É¢„Éº„É´ returns only "mall", dropping Moor/mold/maul/mole). Fix: remove the dedup stage (FTS `content='words'` returns one row per id, so it dedups nothing legitimate), or merge meanings instead of discarding entries.

- **`kanji/[id]` effect refetch loop.** Severity: **High**. `app/word/kanji/[id].tsx:35‚Äì37`. Effect keyed on `[params]`; `useLocalSearchParams()` returns a fresh object every render and `getKanjiById` returns a fresh object, so `setEntry` ‚Üí re-render ‚Üí new `params` ‚Üí re-fire, looping for the screen's lifetime (async, so no crash ‚Äî silent CPU/battery/SQLite drain). Fix: key on `[params.id]` + add an `active`/ignore guard (mirror the sibling `app/word/[id].tsx`).

- **Android/web icons render blank for ~17 of 19 used names.** Severity: **High**. `components/ui/IconSymbol.tsx:9‚Äì28,47`. The MaterialIcons fallback maps only 9 SF Symbol names; unmapped names pass `name={undefined}` ‚Üí empty glyph. Masked by a broken `IconSymbolName` type (`TS2344`/`TS2538`) that gives callers no compile error. iOS unaffected (`.ios.tsx`); Android/web are configured targets. Fix: rekey `MAPPING` by an explicit string-literal union, add all used names, and add `MAPPING[name] ?? 'help-outline'` + a dev warning.

- **Remote AI streams auto-retry 3√ó (paid tokens + stuck spinner).** Severity: **Medium**. `services/queryClient.ts:4` (+ `hooks/useStreamedChat.ts`, `hooks/useStreamedPassage.ts`). `new QueryClient()` has no `defaultOptions`, so RQ's default `retry:3` re-fires thrown stream errors with exponential backoff (~7s) and re-runs server-side generation on genuine HTTP/network errors. (User-cancel via `cancelQueries` is safe; the subscription-throw path burns no tokens but re-shows the paywall.) Fix: `defaultOptions: { queries: { retry: false } }`.

- **Remote chat error leaves an empty assistant bubble + malformed history.** Severity: **Medium**. `hooks/useStreamedChat.ts:44‚Äì66,157‚Äì164`. The placeholder assistant message is only filled when `query.isSuccess`; on a remote error there is no `isError` branch, so the bubble stays blank and `messagesRef` never gets an assistant entry ‚Äî the next turn sends two consecutive user roles. (Local path handles this.) Fix: add a remote-error branch that writes an error message and appends/repairs the assistant entry.

- **Premium AI provider preference silently overridden.** Severity: **Medium**. `providers/SubscriptionProvider.tsx:71‚Äì79`. While entitlement is active, `updateSubscriptionState` unconditionally writes `AI_PROVIDER_TYPE="remote"` on init, restore, paywall close, and every customer-info update, clobbering a premium user's deliberate "local" choice. Fix: only force "remote" on an inactive‚Üíactive transition (or keep just the downgrade branch).

- **`formatJp` ignores the `divider` parameter (missing `break`).** Severity: **Low** (latent). `services/parse.ts:202‚Äì213`. Switch falls through to `default`, always joining with `„ÄÅ`. No current impact (all 5 callers use the default), but any non-default divider would silently break. Fix: add `break;`/`return` per case, or use a lookup map.

- **`findKanji` drops supplementary-plane & Ext-A kanji.** Severity: **Medium**. `services/parse.ts:608‚Äì613`. `char.charCodeAt(0) >= 0x4e00 && <= 0x9faf` reads the high surrogate of Ext-B+ kanji and excludes Ext-A and post-`9FAF` additions; the live seed has 7 supplementary-plane kanji (e.g. ©∏Ω/„Åª„Å£„Åë) whose word-detail kanji section is suppressed and not tappable. Fix: `Array.from(text).filter(c => /\p{Script=Han}/u.test(c))` (wanakana `isKanji` does **not** cover these).

- **`getJpTokens` strips „ÄÖ and halfwidth katakana from suggestion chips.** Severity: **Medium**. `services/parse.ts:540‚Äì566`. `removeJpSymbols` deletes U+3000‚Äì303F (incl. „ÄÖ, U+3005) and U+FF00‚ÄìFFEF (incl. halfwidth katakana), so ÊôÇ„ÄÖÈõ®„ÅåÈôç„Çã fuses ÊôÇ„ÄÖÈõ® into the unrelated word ÊôÇÈõ® (drizzle) and produces a chip that searches the wrong word. Primary search uses raw text, so only the chip feature is affected. Fix: exclude „ÄÖ from the 3000‚Äì303F sweep and don't strip FF61‚ÄìFF9F.

## 2) Database

- **`getWordExamples` text fallback: full scan + per-row `json_each` on ~95% of word opens.** Severity: **High**. `services/database/dictionary.ts:232‚Äì251`. Leading-`%` LIKE OR'd with an `EXISTS(json_each(tokens))` subquery scans all 147,865 example rows (~250ms desktop) on every example-less word's initial detail load, with no cache/gate. Fix: back example lookup with FTS over `japanese_text` (or pre-join via `word_id` at import), or gate behind a cheap existence check. While there, drop the redundant inner `LIKE` OR block (lines 242‚Äì245) ‚Äî it is fully subsumed by the outer `'%word%'`.

- **`searchKanji` full-scans 13,108 kanji per query.** Severity: **Low**. `services/database/kanji.ts:58‚Äì66`. `character LIKE '%q%' OR meanings LIKE '%q%'` bypasses `idx_kanji_character`; `meanings` is JSON so the LIKE also matches structure. Debounced (150ms) and table is small, so impact is modest. Fix: use `character = ?`/prefix for the indexed branch; add an FTS index for meaning search.

- **`SELECT *` pulls the large `search_ngrams` column into results/cache/state.** Severity: **Low**. `services/database/search.ts:138,158,219,228`. `search_ngrams` (avg 64, max 466 chars) and `priority_rank` cross the JSI bridge and get frozen into the 30s cache + React state, unused by the UI; `DBDictEntry` doesn't declare them. Fix: project explicit columns, excluding `search_ngrams` (keep `priority_rank`/`fts_rank` ‚Äî the ORDER BY needs them).

- **Connection PRAGMAs never applied at runtime.** Severity: **Low**. `schema.sql:200‚Äì204` vs `services/database/core.ts` `migrateDbIfNeeded`. `mmap_size`, `cache_size`, `synchronous`, `temp_store` are connection-scoped and don't persist into the seed; the app opens with `mmap_size=0` and a ~2000-page cache on a 288MB read-heavy DB. Fix: re-apply them in `onInit` (mmap + cache are the impactful ones).

- **Non-atomic history writes; missing transaction.** Severity: **Low**. `services/database/history.ts:10‚Äì19,37‚Äì46`. `DELETE`+`INSERT` dedup runs as two un-wrapped statements; a kill between them silently drops the entry. Fix: wrap in `withTransactionAsync`, or add a UNIQUE index + `INSERT ‚Ä¶ ON CONFLICT DO UPDATE`.

- **`practice-sessions.ts` skips `retryDatabaseOperation`.** Severity: **Low**. All 7 functions (`createSession` L37 ‚Ä¶ `deleteSession` L149). Only module violating the CLAUDE.md rule; real BUSY risk is low (single shared connection + busy_timeout). Fix: wrap each call, import from `./utils`.

- **Build importers: words/kanji imports are non-idempotent.** Severity: **Medium** (build tooling). `scripts/import/words-importer.ts:128‚Äì138`, `kanji-importer.ts:84‚Äì95`. No DELETE/UNIQUE, so re-running `db:import` on a populated DB silently duplicates every row (and into `words_fts`). Examples/furigana importers are idempotent. Fix: DELETE-at-start (mirror `resetExamplesTable`) or UNIQUE + ON CONFLICT, or guard a non-empty table.

- **Schema-build swallows trigger-creation failures.** Severity: **Medium** (build tooling). `scripts/import/utils/database.ts:167‚Äì175`. Trigger errors are `console.warn`'d (stale "may not exist yet" rationale ‚Äî triggers run after tables) and never rethrown; `meanings_fts` is populated *solely* by its trigger with no rebuild backstop and is invisible to `getStats`/`db:verify`, so a dropped trigger could ship a silently broken meaning-search index. Fix: rethrow on trigger failure, or replace the hand-rolled splitter with `db.exec(fullSchemaString)`.

- **Lower-impact DB/import items (Low):** declared FOREIGN KEYs never enforced (`foreign_keys` defaults OFF, `schema.sql:6`); `ensureUserDataTables` omits two history indexes (`core.ts:144‚Äì175`); `resetDatabase` swallows errors so the caller's error Alert is dead code (`utils.ts:160‚Äì208`); `updateFTSTables()` redundantly rebuilds `words_fts` and ignores `meanings_fts` (`database.ts:293‚Äì313`); bulk import uses `synchronous=OFF`+`journal_mode=MEMORY` (crash-during-build corruption, disposable artifact); per-entry word+meanings insert not atomic (failure path effectively unreachable); kanji `jis_code` NULL for 62% (hex token vs `/^\d+$/` heuristic; field unused in UI); words importer keeps only `kanji[0]`/`readings[0]` (alternate-form recall tradeoff ‚Äî document it).

## 3) React / UI Correctness

- **`ColorSchemeName` `'unspecified'` type unsafety (theming layer).** Severity: **Medium**. `hooks/useThemeColor.ts:13‚Äì19`, `components/ui/Card.tsx:18‚Äì22`, `app/settings/subscription-info.tsx:21` + 17 `Colors[colorScheme]` sites. In RN 0.85, `?? "light"` widens the index to `string` and doesn't exclude `'unspecified'`, yielding ~21 `TS7053` errors and disabling type-checking on every themed color access (`Colors[scheme]` resolves to `any`). The runtime crash is latent only ‚Äî the app never calls `Appearance.setColorScheme('unspecified')` and native getters return light/dark/null ‚Äî so this is type unsafety + app-wide blast radius, not a live crash. Fix: narrow once at the source: `const scheme: "light" | "dark" = useColorScheme() === "dark" ? "dark" : "light"`.

- **Render-phase mutation in practice session screen.** Severity: **Medium**. `app/practice/[sessionId].tsx`. Two distinct issues, both flagged by `react-hooks/immutability`: (a) `markdownRules` `useMemo` mutates closure counters `paragraphIndex`/`inJapaneseSection` during render (L167,173,186), so under stable-content re-renders or React 19 double-invoke the wrong paragraph plays / wrong play button highlights; (b) `sessionRef.current = session` is assigned during render (L335) feeding the unmount-cleanup DELETE ‚Äî an unmount in the async gap between the stream-complete DB write and the `setSession` render can DELETE a just-persisted passage. Fix: precompute paragraph indexing from the source string (pure render rules); write the ref in an effect (`useEffect(() => { sessionRef.current = session }, [session])`) and gate deletion on a committed flag.

- **`setState`-synchronously-in-effect cluster (lint errors).** Severity: **Low**. `react-hooks/set-state-in-effect` fires at: `app/practice/[sessionId].tsx:81`, `app/practice/index.tsx:48`, `components/ChatView.tsx:72`, `app/word/[id].tsx:93,120`, `app/word/kanji/[id].tsx:125`. These force an extra render and break `yarn lint` (CI gate), but traced behavior is benign (guards prevent loops; reset-to-idle is the correct TTS-completion path). Fix: derive from refs / use functional updaters / move resets into the interval-setup effect.

- **`ChatView` streaming re-render churn.** Severity: **Low**. `components/ChatView.tsx:64‚Äì68` (auto-scroll effect keyed on `messages` array identity fires `scrollToEnd({animated:true})` per token ‚Äî mild stutter, fights manual scroll) and `122‚Äì137` (`renderEmpty` `useCallback` reallocated per token ‚Äî cosmetic, list is never empty during streaming). Fix: key effects on `messages.length`/last-message length; coalesce scroll via rAF and use `animated:false` during streaming.

- **`ListItem` not memoized in a recycled FlashList.** Severity: **Medium**. `components/ListItem.tsx:79` consumed by `app/word/index.tsx:188‚Äì235`. Bare component (no `React.memo`), inline `renderItem` (not `useCallback`), fresh per-row closures, and non-memoized `history.removeItem`/`addKanjiToHistory` cause reconciliation of moderately heavy multi-variant rows on recycle. Fix: `React.memo` + `useCallback(renderItem)` + stable per-id handlers ‚Äî all together (memo alone is defeated by unstable closures). While here, add `getItemType` to the FlashList (heterogeneous history/kanji/search rows) so it recycles within a row kind.

- **`word/[id]` detail load missing cancellation/param refetch.** Severity: **Low**. `app/word/[id].tsx:66‚Äì94`. `[]`-dep effect never refetches on id reuse (latent ‚Äî modal navigation always fresh-mounts) and awaits with no `active` guard (benign on React 19 ‚Äî no unmount warning, no surviving instance to clobber). Fix: add an `active` guard, key on `[params.id]`, or migrate to `useQuery`.

- **`ThemedText` `useMemo` omits `darkColor`/`lightColor` deps.** Severity: **Low** (dormant). `components/ThemedText.tsx:36‚Äì50`. Stale color if those props change while `type` is unchanged ‚Äî but no call site passes them today. Fix: add to deps, or drop the cheap memo.

- **Lower-impact UI items (Low):** root `/` route renders `null` so the `+not-found` recovery link lands on a blank pane (`app/index.tsx`; fix: `<Redirect href="/word" />`); `FuriganaText` wraps per-character so a word's ruby can split across lines on long example text (`components/FuriganaText.tsx:83‚Äì114`); deep-link to `/practice/*` mounts premium screens for non-premium users (UI-only; AI generation is hard-gated ‚Äî add a `_layout` redirect); `paywall.tsx` `handleRestoreFailed` shows a fixed "no subscription" message for all restore errors (`app/paywall.tsx:28‚Äì34`); `IconSymbol` (Android) ignores `weight` and forwards `ViewStyle` where `TextStyle` is expected (`TS2322`).

## Suggested order of work

Start by **unbreaking the test suite** (scoped `tsconfig.test.json` + `@react-native/jest-preset`) and adding a `typecheck` script + minimal CI, so every subsequent fix is regression-gated. Next do the **one-line, high-blast-radius fixes**: narrow `useColorScheme()` once (clears the whole `ColorSchemeName` cluster), set `QueryClient` `retry:false`, and key the `kanji/[id]` effect on `[params.id]`. Then tackle the **core correctness bugs**: the homograph search dedup and `getWordExamples` full scan (the two hottest DB paths), `findKanji`/`getJpTokens` Unicode handling, the IconSymbol mapping, and the remote-chat/subscription error-path defects (`SubscriptionProvider` override, empty bubble, swallowed `generateSpeech` throw). Follow with the **render-phase practice-screen fixes** and the `ListItem`/FlashList memoization. Finish with a **dead-code sweep** (ChatMessage, HighlightText, `getAiExplanation`, the dead remote branch, `reset-project`/`db:generate-readings` scripts) and the remaining low-severity doc/type/theming hygiene, plus the build-tooling import idempotency and trigger-swallow hardening before the next release rebuild.
